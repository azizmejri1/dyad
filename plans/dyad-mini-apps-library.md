# Dyad Mini Apps Library

> Generated by swarm planning session on 2026-03-30

## Summary

Add a built-in library of ready-to-use **feature system mini apps** (i18n, analytics, notifications, search) that Dyad users can integrate into their projects from the Library UI or chat. Each mini app is a hybrid of a **detailed prompt + reference implementation** that the AI adapts to the user's specific project, supporting both React and Next.js templates.

## Problem Statement

Dyad users building web applications repeatedly need the same non-trivial feature systems -- internationalization, analytics tracking, notification systems, search functionality. Today, every user describes these from scratch each time, which:

1. **Wastes LLM tokens** generating the same boilerplate patterns repeatedly
2. **Produces inconsistent results** across sessions -- the AI may wire up i18n differently each time
3. **Takes longer** than necessary for features that have well-known best-practice implementations
4. **Introduces bugs** in complex setup steps (provider wiring, context configuration, locale file structure)

These are not simple UI components -- they are **cross-cutting feature systems** that touch multiple files, require providers/contexts, need configuration, and have specific integration patterns. Getting them right from a prompt alone is error-prone.

## Scope

### In Scope (Phase 1 MVP)

- 4 built-in feature system mini apps: **i18n**, **Analytics**, **Notifications**, **Search**
- Each mini app includes: prompt template + reference implementation code + metadata
- New "Mini Apps" filter tab in the existing Library page
- Mini app detail dialog with description, file list preview, and "Add to App" action
- Chat integration via `/` slash command autocomplete (e.g., `/i18n`, `/analytics`)
- Support for both React and Next.js templates (AI adapts based on detected framework)
- User-initiated only -- no proactive AI suggestions

### Out of Scope (Phase 2 / Follow-up)

- Community sharing / user-created mini apps
- Mini app versioning and update mechanism
- Mini app composition (combining multiple with conflict resolution)
- Visual mini app editor or marketplace
- Proactive AI suggestions ("I see you need i18n, want me to use the mini app?")
- Server-side / backend-only mini apps

## User Stories

- As a Dyad user, I want to browse a library of feature system mini apps so that I can add complex functionality (i18n, analytics) without describing the setup from scratch.
- As a Dyad user, I want to add a mini app to my project with one click so that I get a working, well-structured implementation adapted to my project's framework and existing code.
- As a Dyad user, I want to type `/i18n` in the chat so that the AI uses the pre-built mini app as a starting point and integrates it into my project with the right file structure and wiring.
- As a Dyad user, I want to preview what a mini app includes before adding it so that I understand what files and dependencies will be created.
- As a Dyad user building with Next.js, I want mini apps that work correctly with my framework's conventions (App Router, `"use client"`, etc.) so that I don't get incompatible code.

## UX Design

### User Flow A: Browse and Add from Library

1. User clicks **Library** in the sidebar.
2. Library home page loads with filter tabs: All | Themes | Prompts | Media | **Mini Apps**.
3. User clicks the **Mini Apps** tab. Grid shows mini app cards.
4. Each card displays: icon, name, short description, category badge, "Add to App" button.
5. User clicks a card to open a **detail dialog** showing: full description, list of files that will be created, required dependencies, and a prominent "Add to App" button.
6. User clicks **"Add to App"**. If multiple apps exist, an app picker appears. If one app, it auto-selects.
7. The chat opens with the mini app prompt sent to the AI. The AI generates adapted code, showing file writes in real-time.
8. User reviews the generated files in the preview panel and can iterate via chat.

### User Flow B: Add from Chat via Slash Command

1. User is in an active chat session with an app selected.
2. User types `/` in the chat input. Autocomplete menu appears with a "Mini App" category.
3. User types `/i18n` or `/analytics`. Autocomplete filters to matching mini apps.
4. User selects the mini app. A mention node is inserted into the input (styled with a distinct color).
5. User can add context: `/i18n -- I need English, French, and Arabic with RTL support`.
6. On send, the system resolves the mini app reference, injects the prompt + reference code into the AI context.
7. The AI generates adapted code. User sees file writes streaming in real-time.

### Key States

- **Default (Mini Apps tab)**: Grid of mini app cards with search bar active. Cards show icon, title, one-line description, category badge.
- **Detail Dialog**: Full description, file list preview ("This will create: `src/i18n/`, `src/hooks/useTranslation.ts`, `src/components/LanguageSwitcher.tsx`"), dependency list, "Add to App" button.
- **Loading**: After clicking "Add to App", chat shows the AI generating files with the standard streaming animation.
- **Empty Search**: "No mini apps match your search." with suggestion to clear filter.
- **No App Selected**: When adding from Library without an active app: "Select an app first, then add mini apps to it." with app picker.
- **Error**: Toast notification if file write fails, with actionable message.

### Interaction Details

- Mini app cards are **read-only** (no edit/delete actions, unlike prompts/themes). No action icons in card corners.
- Cards are fully clickable (open detail dialog). The "Add to App" button is a shortcut that skips the dialog.
- In chat autocomplete, mini apps get a distinct badge color (green "Mini App" pill) to distinguish from prompts and skills.
- After adding, the chat shows a summary of what was created. The preview panel refreshes to show new files.
- Search bar placeholder updates to include "mini apps" when on the Mini Apps tab.

### Accessibility

- Filter tabs inherit existing `aria-pressed` button semantics from `LibraryFilterTabs`.
- Mini app cards are focusable (`tabIndex={0}`) with Enter/Space to open detail dialog.
- "Add to App" button is a separate tab stop within each card.
- Detail dialog traps focus and is dismissible with Escape (Shadcn dialog pattern).
- Badge colors must meet WCAG AA contrast ratio (4.5:1) in both light and dark themes.
- Chat autocomplete items have clear labels for screen readers.

## Technical Design

### Architecture

Mini apps are **static data** shipped with the Dyad application, defined in TypeScript. Each mini app consists of:

1. **Metadata**: id, title, description, category, icon, tags, dependencies
2. **Prompt template**: A detailed prompt that instructs the AI how to implement the feature
3. **Reference implementation**: Actual code files that serve as the starting point / reference for the AI

When a user selects a mini app, Dyad constructs a chat message that includes:
- The mini app's prompt template
- The reference implementation code (as context)
- Instructions for the AI to adapt the code to the user's project (detected framework, existing files, styling)

The AI then generates the adapted code using its standard file-write tools (`<dyad-write>`, `<dyad-add-dependency>`, etc.).

### Mini App Data Format

```typescript
// src/shared/mini-apps.ts

export interface MiniApp {
  id: string;                    // e.g., "i18n"
  title: string;                 // "Internationalization (i18n)"
  description: string;           // Short description for cards
  longDescription: string;       // Full description for detail dialog
  category: MiniAppCategory;     // "i18n" | "analytics" | "notifications" | "search"
  icon: string;                  // Lucide icon name, e.g., "Languages"
  tags: string[];                // For search: ["translation", "locale", "multilingual"]
  dependencies: string[];        // npm packages: ["i18next", "react-i18next"]
  files: MiniAppFile[];          // Reference implementation files
  prompt: string;                // AI prompt template
}

export interface MiniAppFile {
  path: string;                  // Relative path, e.g., "src/i18n/index.ts"
  description: string;           // "i18n configuration and initialization"
  content: string;               // Reference implementation code
}

export type MiniAppCategory =
  | "i18n"
  | "analytics"
  | "notifications"
  | "search";

export const miniAppsData: MiniApp[] = [
  // ... 4 built-in mini apps
];
```

### Data Model Changes

**No database changes for Phase 1.** Mini apps are static TypeScript data shipped with the app, similar to `src/shared/templates.ts`. This means:
- No migration needed
- Easy version control and code review of mini app prompts/reference code
- Updates ship with each Dyad release
- Zero IPC overhead (data available directly in renderer)

### API Changes

**New hook** (`src/hooks/useMiniApps.ts`):

  category?: MiniAppCategory;
  search?: string;

**Chat integration** follows existing pattern:
- `replaceMiniAppReference(message: string)` in chat stream handlers -- same pattern as `replaceSlashSkillReference()`
- Expands `/i18n` to the full prompt + reference code block

**No new IPC contracts needed** for Phase 1 since all data is static and lives in the renderer process.

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| AI produces inconsistent code across frameworks | Medium | High | Include framework detection in prompt. Test each mini app against both React and Next.js. Include explicit instructions for Next.js conventions (`"use client"`, App Router). |
| Reference code becomes stale as dependencies update | Medium | Medium | Pin dependency versions in mini app definitions. Review quarterly. Keep reference code minimal -- rely on the AI to use latest patterns. |
| Mini app integration conflicts with existing project files | Medium | High | AI prompt includes instructions to check for existing files before writing. Chat shows file list before generating. User can review and iterate. |
| Maintenance burden of 4 mini apps x 2 frameworks | Low | Medium | Start with prompt-heavy approach (less reference code to maintain). Reference code is a guide, not verbatim copy. |
| Users expect more mini apps than we provide | Low | Low | Set clear expectations in UI ("4 built-in feature systems"). Phase 2 enables community contributions. |
| Chat autocomplete becomes crowded with mini apps + prompts + skills | Low | Low | Distinct badge colors and category headers in autocomplete menu. Mini apps are a small fixed set (4 items). |

## Open Questions

1. **Naming**: Is "Mini Apps" the right user-facing term? Alternatives: "Starters", "Add-ons", "Feature Kits". Should be tested with users but "Mini Apps" is clear enough for v1.

2. **Post-integration follow-up**: After adding a mini app, should Dyad automatically open a follow-up prompt (e.g., "The i18n system is set up. Which languages do you need?") or let the user drive?

3. **Dependency conflicts**: If a mini app requires a package the user already has at a different version, how should the AI handle it? Current recommendation: AI checks `package.json` and skips already-installed packages.

4. **Mini app updates**: If we improve a mini app in a future Dyad release, should users who already added it be notified? Phase 1 answer: No. This is a Phase 2 concern.

## Decision Log

| Decision | Rationale |
|----------|-----------|
| **Hybrid approach** (prompt + reference code) over pure prompt or pure file copy | User preference. Reference code gives AI a concrete starting point for consistent output. Prompt allows framework adaptation. Better than file copy (too brittle) or pure prompt (too inconsistent for complex features). |
| **Both React and Next.js** from day one | User preference. With the hybrid approach, the AI can adapt reference code to the detected framework. The prompt includes framework-specific instructions. |
| **User-initiated only** (no proactive suggestions) | User preference. Simpler to implement. Avoids system prompt complexity. Users discover mini apps through Library UI and `/` commands. |
| **Feature systems, not UI components** | User preference. i18n, analytics, notifications, search are cross-cutting concerns that are complex to set up from scratch. Simple UI components are easy enough to generate from a chat prompt. |
| **Static TypeScript data, no database** | Engineering recommendation. No migration, easy version control, ships with updates. Phase 2 (community sharing) would add DB storage. |
| **New filter tab in existing Library** | UX recommendation. Consistent with existing patterns (themes, prompts, media). Minimal new UI surface. |
| **4 mini apps for v1** (i18n, analytics, notifications, search) | User-specified. These are complex feature systems that every app benefits from and are tedious to set up correctly from scratch. |

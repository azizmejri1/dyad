# Automated Database Migrations via Schema Diff

> Generated by swarm planning session on 2026-03-30

## Summary

Add a "Generate Migration" button to the Publish panel that introspects the Neon development and production branch schemas, generates a SQL migration from the diff using drizzle-kit, and lets the user review and apply the migration with one click. This closes the gap between prototyping (where the AI modifies the dev database directly) and production deployment (where schema changes currently don't propagate).

## Problem Statement

When a Dyad user builds an app with a Neon database, the AI agent modifies the development database schema via direct SQL (`execute_neon_sql`). When the user deploys to production via Vercel, there is no mechanism to propagate those schema changes to the production Neon branch. The development and production databases silently diverge, and the deployed app breaks because its code expects a schema that doesn't exist in production. This is a deployment-blocking, data-integrity problem that hits every Neon user who iterates on their schema and then ships.

## Scope

### In Scope (MVP)

- Schema introspection of both Neon branches (development and production/default) using drizzle-kit
- SQL diff generation from the schema comparison
- User review of generated migration SQL with syntax highlighting
- Destructive operation detection (DROP TABLE, DROP COLUMN, ALTER COLUMN TYPE) with red warnings and extra confirmation dialog
- One-click migration execution against the production branch (wrapped in a transaction)
- "Generate Migration" button on the Publish panel, visible when a Neon project is connected
- Non-blocking warning on Vercel deploy section if schemas appear out of sync
- Explicit user-triggered diff only (no auto-detection on panel open)
- Empty diff state ("schemas in sync") and first-time setup state ("set up production schema")
- Dev schema validation (at least one table must exist before diffing)
- Defined error states for all failure modes (auth expired, endpoint suspended, introspection fails, SQL invalid, network timeout, partial failure)

### Out of Scope (Follow-up)

- Auto-migration as part of Vercel deploy pipeline (pre-deploy hook)
- Migration rollback support
- Migration history / audit log
- Supabase equivalent (Neon-only for MVP)
- Data migrations (DML, not just DDL)
- Preview branch migration
- Editable migration SQL (users must fix dev schema and regenerate)
- Auto-detect drift on panel open
- Rename conflict resolution UI (column renamed vs. dropped + added)
- Consolidation with existing PortalMigrate component (coexist for now)

## User Stories

- As a Dyad user with a Neon-backed app, I want to see what schema differences exist between my development and production databases so that I understand what will change before I deploy.
- As a Dyad user about to deploy, I want to generate and review a SQL migration from the dev/prod schema diff so that I can approve it before it touches production data.
- As a Dyad user, I want to apply the approved migration to my production Neon branch with one click so that my deployed app's code matches its database schema.
- As a Dyad user, I want to be told "no changes detected" when my schemas are already in sync so that I don't waste time worrying about missing migrations.
- As a Dyad user, I want destructive operations (DROP TABLE, DROP COLUMN) to be clearly flagged with warnings so that I don't accidentally delete production data.
- As a Dyad user deploying for the first time, I want the migration flow to set up my production database schema from scratch so that my first deployment works.

## UX Design

### User Flow

1. User has been developing with Neon -- AI agent modifies development branch schema via direct SQL.
2. User navigates to the Publish panel.
3. User sees a "Database Migration" card (visible because `app.neonProjectId` is set and a development branch exists).
4. User clicks "Generate Migration."
5. Button enters loading state with "Analyzing schemas..." message (may take 3-8 seconds if Neon endpoint is cold -- show "Waking up database..." for cold starts).
6. System introspects both branches via drizzle-kit, computes the diff, displays the resulting SQL.
7. User reviews the migration:
   - Statements grouped by type: additions (green), alterations (yellow), deletions (red).
   - Summary line: "3 tables added, 1 column altered, 1 table dropped."
   - Syntax-highlighted SQL in a scrollable code block (max-h-64, expandable for large diffs).
   - Destructive operations flagged with red highlight and warning icon.
8. User clicks "Apply Migration to Production."
   - If migration contains destructive operations: extra confirmation dialog ("This migration will drop table X. Are you sure?") before execution.
   - If non-destructive: single click applies.
9. System executes migration SQL against production branch (wrapped in a single Postgres transaction).
10. Success toast + inline confirmation. Migration card resets to idle state.
11. User proceeds with Vercel deployment (schema now in sync).

**Alternative: No changes detected**
- User clicks "Generate Migration."
- System introspects, finds schemas identical.
- Shows calm message: "Your production schema is already up to date."

**Alternative: First-time setup**
- Production branch has no tables.
- Messaging adjusts: "This will set up your production database schema for the first time."
- Same review and apply flow.

### Key States

- **Idle**: Card with "Generate Migration" button. Shows branch names being compared (dev branch name vs. production/default).
- **Loading**: Button disabled with spinner. Message: "Analyzing schemas..." or "Waking up production database..." for cold endpoints.
- **Diff Display**: Scrollable SQL code block with syntax highlighting. Summary counts. Destructive ops in red. "Apply Migration to Production" button.
- **Empty Diff**: "Your production schema is already up to date." No action button. Checkmark icon.
- **Error**: Inline error with specific message (auth expired, connection failed, introspection failed, etc.) and "Retry" button. Toast for transient errors.
- **Applying**: "Apply" button shows spinner, "Applying migration...". Disabled to prevent double-click.
- **Success**: Toast + inline success: "Migration applied successfully. X statements executed."
- **Failed**: Inline error with failure details. Since migration is wrapped in a transaction, it either fully succeeds or fully rolls back -- communicate this clearly: "Migration failed and was rolled back. No changes were made to your production database."

### Interaction Details

- Two-step interaction: Generate (read-only, safe) then Apply (consequential). Never combine into single click.
- Destructive operations trigger an extra confirmation dialog before Apply executes.
- Scrollable diff area with max-height constraint. Summary line visible without scrolling.
- Branch names displayed using existing Badge component with `getBranchBadgeVariant` styling.
- Migration card positioned on the Publish panel alongside (not blocking) the Vercel deploy section.
- If schemas are out of sync and user has already generated a diff, show a subtle warning indicator on the Vercel deploy section: "Database schema may be out of sync."

### Accessibility

- Generate and Apply buttons focusable and activatable via Enter/Space.
- Diff code block scrollable via keyboard.
- Destructive operation warnings include `aria-label` text (not color-only).
- Loading state uses `aria-busy="true"`.
- Confirmation dialog is focus-trapped with Escape to cancel.
- Code block has sufficient color contrast in both light and dark modes.
- Touch targets meet 44x44px minimum.

## Technical Design

### Architecture

Use drizzle-kit CLI with temporary configs, following the existing pattern from `portal_handlers.ts`:

1. Retrieve connection URIs for both dev and production branches via Neon API (`neon:get-connection-uri`).
2. Write a temporary `drizzle.config.ts` pointing at the dev branch connection URI.
3. Run `drizzle-kit introspect --config=<tmp>/drizzle-dev.config.ts` to generate Drizzle schema files reflecting the dev database.
4. Write a second temporary `drizzle.config.ts` pointing the introspected dev schema at the production branch connection URI.
5. Run `drizzle-kit push --verbose` (or `--dry-run` if available) against production, capturing the planned SQL statements. Pipe `n\n` to stdin to abort actual execution.
6. Parse the captured SQL output as the migration diff.
7. On user confirmation, execute the SQL against production via `executeNeonSql` wrapped in a single transaction (`BEGIN; ... COMMIT;`).
8. Clean up temporary directories.

**Fallback**: If drizzle-kit output parsing proves unreliable in the spike, fall back to `pg_dump --schema-only` diffing or direct `information_schema` comparison using the existing queries in `neon_context.ts`.

### Components Affected

- `src/ipc/handlers/migration_handlers.ts` (NEW) -- IPC handlers for migration:generate-diff and migration:apply
- `src/ipc/types/migration.ts` (NEW) -- Zod contracts for migration IPC
- `src/components/MigrationPanel.tsx` (NEW) -- UI component for migration button, diff display, apply
- `src/ipc/handlers/neon_handlers.ts` -- Extract `getProductionBranchId()` utility (production = default branch)
- Handler registration file -- Register new migration handlers
- Publish panel component -- Add MigrationPanel when Neon project is connected
- Vercel deploy section -- Add "schema out of sync" warning indicator

### Data Model Changes

No persistent schema changes required for Dyad's internal SQLite database. The migration flow is stateless -- diff is generated on demand and discarded after execution. Migration status is held in React component state.

**Optional future addition**: `lastMigrationTimestamp` field on the apps table for showing "schema may be out of sync" indicators.

### API Changes

**`migration:generate-diff`**
- Input: `{ appId: string }`
- Output: `{ sql: string; statements: string[]; hasChanges: boolean; summary: { added: string[]; altered: string[]; dropped: string[] } }`
- Behavior: Introspects both branches, generates diff SQL, returns structured result.

**`migration:apply`**
- Input: `{ appId: string; sql: string }`
- Output: `{ success: boolean; error?: string; statementsExecuted: number }`
- Behavior: Wraps SQL in a transaction, executes against production branch via `executeNeonSql`. Includes staleness mitigation (re-validate diff before applying).

**Connection string resolution**: Use the Neon API to resolve the production (default) branch connection string from the project ID. Do NOT read from Vercel env vars or ask the user to paste it.

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
| --- | --- | --- | --- |
| drizzle-kit CLI output parsing is fragile (format changes across versions) | High | High | Pin drizzle-kit version strictly; smoke test against known output; clear error if parsing fails |
| Destructive migrations applied to production accidentally | Medium | Critical | Destructive ops flagged with red warnings; extra confirmation dialog required; all migrations wrapped in transaction |
| Neon endpoint cold start causes timeout during introspection | Medium | Medium | Retry logic; contextual "Waking up database..." loading message; consider pre-warming via HTTP query |
| Temp directory cleanup fails, leaving connection strings on disk | Medium | Medium | try/finally cleanup; OS temp dir with short TTL; never log passwords |
| drizzle-kit introspect misses Postgres features (extensions, RLS policies, triggers, custom types) | Medium | Medium | Document limitations; for Dyad-created schemas (simple tables), low risk; validate introspected schema is non-empty |
| Electron child process spawning breaks in packaged builds | Medium | High | Follow existing portal_handlers.ts pattern; test in packaged builds during Phase 3 |
| Stale migration applied (dev schema changed between generate and apply) | Low | High | Re-validate diff hash before applying; timeout generated migrations after 10 minutes |
| Rename ambiguity (drop+add vs rename) | Medium | Low | Accept drizzle-kit's interpretation; document limitation; no custom rename detection for v1 |

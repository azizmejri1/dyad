# Remotion Motion Graphics Support

> Generated by swarm planning session on 2026-03-30

## Summary

Extend Dyad with a Pro feature that lets existing web-app builders generate animated motion graphics (explainer videos, product demos, social media content) using Remotion, a React-based programmatic video framework. Users describe what they want in chat, the AI generates Remotion composition code, a live preview plays in the preview panel, and the final video renders to MP4/WebM via ffmpeg.

## Problem Statement

Dyad users building web apps occasionally need animated video content — promotional clips, product walkthroughs, animated explainers, social media ads — but must leave Dyad entirely to create them. There is no AI-native tool that lets developers describe a motion graphic in natural language and get a working, renderable video project using standard, ownable code. By adding Remotion support, Dyad extends its AI-assisted creation model from web apps to video, serving an adjacent need for the same user base.

## Scope

### In Scope (MVP)

- **Remotion project template**: New scaffold template that creates a working Remotion project with example compositions
- **AI code generation for Remotion**: System prompt engineering so the AI generates valid Remotion composition code (constrained API surface)
- **Live preview**: Remotion Player embedded via the existing Vite dev server and iframe preview panel, with play/pause/scrub controls and aspect ratio presets
- **Video rendering/export**: "Render Video" action that shells out to `npx remotion render`, with progress tracking, cancellation, and output file management
- **ffmpeg detection & install guide**: Check for ffmpeg at render time, show platform-specific install instructions if missing
- **Pro gating**: Feature gated behind Dyad Pro, following existing `isDyadProEnabled` patterns
- **First-run experience**: Curated starter compositions (3-5 archetypes) and example prompts to reduce blank-canvas drop-off
- **Error handling**: Compilation errors shown inline with "Fix this" button; render errors with ffmpeg stderr details

### Out of Scope (Follow-up)

- Audio support (`<Audio>` tags, music sync, voiceover)
- Visual timeline / keyframe editor
- Multi-scene / multi-composition projects with transitions
- Cloud rendering (Remotion Lambda)
- Asset management (fonts, media clips, stock footage)
- Bundling ffmpeg with Dyad
- Remotion project import (detecting existing Remotion projects)
- Deployment / hosting of rendered videos

## User Stories

- As a **web-app developer using Dyad**, I want to describe an animated promo video for my app and get a working Remotion project so that I can create marketing content without learning animation programming.
- As a **Dyad Pro user**, I want to preview my motion graphic in real time as the AI iterates on it so that I get the same immediate feedback loop I have with web app development.
- As a **Dyad user with a finished composition**, I want to render it to an MP4 file so that I can share or upload it without leaving Dyad.
- As a **first-time video creator in Dyad**, I want to start from an example composition so that I have a concrete starting point rather than a blank canvas.

## UX Design

### User Flow

1. User opens Dyad, clicks "New App," and selects the **"Motion Graphics (Remotion)"** template card (marked with a Pro badge)
2. User names the project; Dyad scaffolds a Remotion project with a starter composition
3. User lands on the familiar **chat + preview** layout
4. User describes the desired animation in chat (e.g., "Create a 10-second intro with my logo sliding in from the left, then text fading in")
5. AI generates/modifies Remotion composition code; streaming diffs appear as usual
6. **Preview panel** shows the Remotion Player with play/pause, scrub bar, and frame counter — updates live as AI writes code
7. User iterates via chat ("Make the logo bounce instead of slide")
8. When satisfied, user clicks **"Render Video"** in the preview toolbar
9. **Export panel** shows format (MP4/WebM), resolution (1080p default), and an "Advanced" section
10. User clicks "Render" — progress bar with ETA and cancel button appears
11. On completion, toast notification with "Open file" / "Open folder" actions

### Key States

- **Template selection**: "Motion Graphics (Remotion)" card with Pro badge and brief description. Free users see it but hit a Pro gate on creation.
- **Empty/first-run**: Preview panel shows a sample composition auto-playing with a pre-filled chat suggestion: "Try: Make a 5-second title card that says 'Hello World' with a fade-in"
- **Loading**: Spinner with "Preparing preview..." while Vite dev server starts
- **Active preview**: Remotion Player with play/pause (Space), frame step (arrow keys), scrub bar, frame counter. Aspect ratio presets (16:9, 9:16, 1:1) replace device-size toggles.
- **Compilation error**: Inline error below the player with "Fix this" button that auto-populates the chat with the error context
- **ffmpeg not found** (at render time only): Non-modal inline banner with OS-specific install command (brew/winget/apt) and a "Re-check" button
- **Rendering in progress**: Determinate progress bar with frame count (e.g., "Frame 120/360"), estimated time remaining, and cancel button. Non-blocking — user can continue editing.
- **Render complete**: Toast with file path and "Open in Finder/Explorer" action
- **Render failed**: Expandable error details with ffmpeg stderr output

### Interaction Details

- **Play/Pause**: Spacebar when preview panel is focused
- **Frame navigation**: Left/Right arrow keys for frame-by-frame stepping
- **Seek**: Click/drag on timeline scrubber (44px minimum touch target)
- **Hot-reload**: Code changes update the preview without resetting the playhead position (capture frame, HMR, restore via `seekTo()`)
- **Render feedback**: Persistent progress indicator in the sidebar (non-modal), similar to image generation progress pattern

### Accessibility

- Full keyboard operation: Space (play/pause), arrows (frame step), Home/End (jump to start/end), Tab (navigate controls)
- ARIA labels on all player controls: "Play/Pause button", "Timeline scrubber, currently at 3.2 seconds of 10 seconds"
- `prefers-reduced-motion`: Default to paused state, require explicit play
- Render progress announced via `aria-live` regions
- Focus rings on all interactive elements
- WCAG AA contrast for timeline controls and frame counter

## Technical Design

### Architecture

Remotion projects are managed as a new app type within Dyad's existing app management infrastructure. The scaffold template creates a standard Remotion + Vite project. The Vite dev server (already managed by Dyad's process manager) serves a page with `@remotion/player` for preview. Rendering shells out to `npx remotion render` as a child process with progress parsing.

```
[Chat Panel] → AI generates Remotion code → [Files on disk]
                                                    ↓
[Preview Panel (iframe)] ← Vite dev server ← @remotion/player
                                                    ↓
[Render Action] → npx remotion render (child process) → MP4/WebM file
                         ↓
              ffmpeg (user-installed)
```

### Data Model Changes

**`apps` table** — add column:
```sql
ALTER TABLE apps ADD COLUMN project_type TEXT DEFAULT 'web-app';
-- Values: 'web-app' | 'remotion'
```

**New `renders` table** (optional, for render history):
```sql
CREATE TABLE renders (
  id TEXT PRIMARY KEY,
  app_id TEXT NOT NULL REFERENCES apps(id),
  status TEXT NOT NULL DEFAULT 'queued', -- queued | rendering | complete | failed
  format TEXT NOT NULL, -- mp4 | webm
  resolution TEXT, -- e.g., '1920x1080'
  output_path TEXT,
  started_at INTEGER,
  completed_at INTEGER,
  error_message TEXT
);
```

**Video project metadata**: Aspect ratio, frame rate, and duration stored as defaults in the scaffold template's `remotion.config.ts`, not in the database. The AI can modify these in code.

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
| --- | --- | --- | --- |
| LLM generates broken Remotion code | High | High | Constrain API surface, curated few-shot examples, structured error interception with auto-fix |
| React version incompatibility (Remotion vs React 19) | Medium | High | Verify early in Phase 1; scaffold template pins compatible React version independently |
| Remotion licensing creates legal issues | Medium | High | Legal review in Phase 1; Pro-only gating limits exposure; add license notice in template |
| ffmpeg install friction causes user drop-off | Medium | Medium | OS-specific one-liner commands, "Re-check" button, only prompt at render time (not preview) |
| Rendering consumes excessive CPU/memory | Medium | Medium | Run in separate process, show resource warning for long compositions, allow cancellation |
| Orphaned ffmpeg/render processes | Low | Medium | Robust process tracking via existing `runningApps` pattern, cleanup on app quit |
| Rendered video files consume disk space | Low | Low | Show file sizes in render history, provide delete action |

## Open Questions

- **Remotion license for scaffold template**: Does distributing a scaffold template that includes Remotion as a dependency require Dyad (the company) to hold a Remotion license? Legal review needed before Phase 1 begins.
- **Optimal LLM model/prompt strategy**: Which LLM providers produce the most reliable Remotion code? Needs experimentation during Phase 2. May need provider-specific prompt tuning.
- **Render output location**: Should renders go to a project-level `renders/` directory (cleaner) or the system Downloads folder (more discoverable)? Decide during Phase 4 UX implementation.

## Decision Log

| Decision | Reasoning |
| --- | --- |
| Include rendering in v1 | Preview-only feels like a tech demo, not a product feature. The complete prompt-to-video workflow is the core value proposition. |
| Target existing Dyad web-app builders | Adjacent need for the same user base. Avoids the risk of building for a new segment with uncertain demand. |
| Require user to install ffmpeg | Aligns with "Bridge, Don't Replace" principle. Avoids bloating Dyad's download size by 100-150MB for all users. |
| Pro feature | Matches positioning of other advanced capabilities (local agent, visual editing). Funds ongoing maintenance. |
| Standalone project type (not subsystem) | Clean mental model, avoids Vite config conflicts, consistent with existing template pattern. |
| Chat-only paradigm (no visual timeline) | Chat-driven creation is Dyad's differentiator. A timeline editor is a different product with massive engineering cost. |
| Audio explicitly out of scope | Different problem domain, scope landmine. Excluded from AI's Remotion API surface to prevent broken compositions. |
| `@remotion/player` via Vite (not Remotion Studio) | Remotion Studio has its own UI we can't control. Player component served via Vite reuses existing dev server infrastructure. |
| Constrained Remotion API surface | Narrow, reliable generation beats broad but brittle. Improves AI output quality significantly. |
| Aspect ratio set at project creation | Changing aspect ratio after generation breaks layouts. First-class decision, not adjustable without re-generation. |
